# Wind Server UI - Angular Frontend Instructions

## Project Overview
This is the Angular frontend for the Wind Server application - a game server management platform. The UI communicates with a Python FastAPI backend and provides authentication, registration, and blog functionality.

## Tech Stack
- **Framework**: Angular 19.2 (standalone components)
- **Language**: TypeScript 5.7
- **Styling**: SCSS
- **Testing**: Jasmine & Karma
- **HTTP**: Angular HttpClient with interceptors

## Project Structure
```
src/app/
├── components/         # Feature components (home, login, register)
├── services/          # Business logic (auth, theme)
├── guards/            # Route guards (auth protection)
├── interceptors/      # HTTP interceptors (auth headers)
├── app.routes.ts      # Route definitions
└── app.config.ts      # Application configuration
```

## Architecture Patterns

### Standalone Components
All components use Angular's standalone component API (no NgModules). Import dependencies directly in component decorators.

### Dependency Injection
- Use `inject()` function for dependency injection in modern Angular style
- Services are provided at root level by default
- Interceptors registered via `provideHttpClient(withInterceptors([...]))`

### Routing & Guards
- Route definitions in `app.routes.ts`
- Auth guard protects authenticated routes
- Use Angular Router for navigation

## Code Generation Guidelines

### Output Formatting
- No emojis in code, comments, or responses
- No ASCII art or decorative symbols
- No explanatory comments about changes (e.g., `// I changed this to fit your choices!`)
- Code should be clean and self-documenting
- Comments should explain "why" not "what was changed"

### Test Case Management
- Always check for existing test files when modifying code
- Update related tests when changing component/service logic
- Ensure tests remain valid after code changes
- Run tests to verify changes don't break existing functionality

## Coding Conventions

### File Naming
- Components: `feature-name.component.ts`
- Services: `service-name.service.ts`
- Guards: `feature.guard.ts`
- Interceptors: `feature.interceptor.ts`

### Component Structure
```typescript
import { Component } from '@angular/core';
import { CommonModule } from '@angular/common';

@Component({
  selector: 'app-feature-name',
  standalone: true,
  imports: [CommonModule, /* other imports */],
  templateUrl: './feature-name.component.html',
  styleUrl: './feature-name.component.scss'
})
export class FeatureNameComponent {
  // Component logic
}
```

### TypeScript Style
- Use strict typing - avoid `any`
- Prefer `const` over `let`
- Use arrow functions for callbacks
- Use optional chaining (`?.`) and nullish coalescing (`??`)
- Interface names without "I" prefix (e.g., `User` not `IUser`)

### rxjs Patterns
- Use `async` pipe in templates when possible
- Unsubscribe from observables in `ngOnDestroy`
- Prefer observables over promises for HTTP calls

### SCSS Organization
- Use component-scoped styles (`.component.scss` files)
- Follow BEM naming convention for CSS classes
- Use theme service for consistent theming

## API Integration

### Base URL
API base URL should be configurable (currently `http://localhost:8000`)

### Authentication
- JWT tokens stored in localStorage
- Auth interceptor automatically adds `Authorization: Bearer <token>` headers
- Auth service manages login/logout/registration
- Auth guard protects routes requiring authentication

### HTTP Patterns
```typescript
// In a service
constructor(private http = inject(HttpClient)) {}

getData() {
  return this.http.get<ResponseType>(`${API_URL}/endpoint`);
}
```

## Development Guidelines

### Commands
- `npm start` - Start dev server (http://localhost:4200)
- `npm run build` - Production build
- `npm test` - Run tests
- `npm run watch` - Watch mode build

### Component Creation
- Always use Angular CLI: `ng generate component components/feature-name`
- Include tests for all components
- Use OnPush change detection when appropriate

### State Management
- Currently using service-based state (no NgRx/Signals)
- Services act as state managers (e.g., AuthService)
- Consider adding signals for reactive state in future

### Error Handling
- Display user-friendly error messages
- Log errors to console in development
- HTTP errors should be caught and handled gracefully

### Accessibility
- Use semantic HTML elements
- Include ARIA labels where needed
- Ensure keyboard navigation works
- Maintain proper heading hierarchy

## Security Best Practices
- Never store sensitive data in localStorage except JWT tokens
- Sanitize user input
- Use Angular's built-in XSS protection
- Auth guard prevents unauthorized access
- Tokens automatically included in requests via interceptor

## Testing
- Write unit tests for all components and services
- Test authentication flows
- Mock HTTP calls in tests
- Aim for meaningful coverage, not just high percentages
